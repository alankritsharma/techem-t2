# Demo Contract Reference

This document describes the JSON artifacts generated by:

```bash
python scripts/run_demo_export.py
```

The backend remains the forecasting authority. These files are meant to support
fake-login landlord and tenant flows in the frontend without introducing real
authentication yet.

## Local Demo API

The generated files can be served through the thin FastAPI layer:

```bash
python3.11 scripts/run_demo_api.py --output-dir outputs/demo --host 127.0.0.1 --port 8000
```

Endpoints:

- `POST /auth/login`
- `GET /auth/me`
- `GET /landlord/dashboard`
- `GET /landlord/properties`
- `GET /properties/{property_id}`
- `GET /properties/{property_id}/daily`
- `GET /properties/{property_id}/monthly`
- `GET /tenant/dashboard`
- `GET /tenant/daily`
- `GET /tenant/monthly`
- `GET /health`

Authentication is demo-only:

- login validates against `demo_users.json`
- API returns a fake bearer token of the form `demo-token:<user_id>`
- no database, no refresh tokens, no production security model

## Canonical IDs

- `property_id`: backend property identifier derived from `dataset/property_*.csv`
- `unit_id`: backend unit identifier derived from `property_id + unit_number`
- `landlord_id`: demo landlord identifier such as `landlord_01`
- `tenant_id`: demo tenant identifier such as `tenant_property_1_unit_1`

## Date and Number Conventions

- Dates are ISO-8601 strings
- All numeric values are raw numbers
- Currency/unit formatting belongs in the frontend

## Generated Files

### `demo_users.json`

Flattened fake login registry for both landlords and tenants.

Required fields:

- `user_id`
- `role`
- `display_name`
- `username`
- `email`
- `password`
- landlord-specific assignment fields or tenant-specific assignment fields

### `landlords.json`

Landlord registry with assigned properties.

Required fields:

- `landlord_id`
- `display_name`
- `username`
- `email`
- `password`
- `property_ids`

### `tenants.json`

Tenant registry with unit assignments.

Required fields:

- `tenant_id`
- `display_name`
- `username`
- `email`
- `password`
- `property_id`
- `unit_id`
- `landlord_id`

### `property_registry.json`

Metadata for the 20 demo properties.

Required fields:

- `property_id`
- `property_number`
- `display_name`
- `city`
- `zipcode`
- `energy_source`
- `living_space_m2`
- `unit_count`
- `room_count`
- `landlord_id`

### `unit_registry.json`

Metadata for unit-level tenant assignments.

Required fields:

- `unit_id`
- `property_id`
- `display_name`
- `unit_number`
- `living_space_m2`
- `room_count`
- `tenant_id`
- `landlord_id`

### `daily_forecast.json`

Full unit-level daily forecast rows for the portfolio.

Core fields:

- `date`
- `property_id`
- `unit_id`
- `predicted_energy_kwh`
- `predicted_cost_eur`
- `predicted_co2_kg`
- `outside_temp_c`

### `monthly_summary.json`

Unit-level monthly totals from the forecasting backend.

Core fields:

- `property_id`
- `unit_id`
- `series_id`
- `month`
- `predicted_energy_kwh`
- `predicted_cost_eur`
- `predicted_co2_kg`

### `annual_summary.json`

Unit-level annual totals from the forecasting backend.

Core fields:

- `property_id`
- `unit_id`
- `series_id`
- `year`
- `predicted_energy_kwh`
- `predicted_cost_eur`
- `predicted_co2_kg`

### `property_summaries.json`

Property-level totals over the exported forecast horizon.

Core fields:

- `property_id`
- `landlord_id`
- `display_name`
- `predicted_energy_kwh`
- `predicted_cost_eur`
- `predicted_co2_kg`
- `as_of_date`

### `unit_summaries.json`

Unit-level totals over the exported forecast horizon.

Core fields:

- `property_id`
- `unit_id`
- `tenant_id`
- `landlord_id`
- `predicted_energy_kwh`
- `predicted_cost_eur`
- `predicted_co2_kg`
- `as_of_date`

### `landlord_summaries.json`

Landlord dashboard totals across assigned properties.

Core fields:

- `landlord_id`
- `display_name`
- `property_count`
- `unit_count`
- `predicted_energy_kwh`
- `predicted_cost_eur`
- `predicted_co2_kg`
- `as_of_date`

### `tenant_summaries.json`

Tenant dashboard totals for the assigned unit.

Core fields:

- `tenant_id`
- `property_id`
- `unit_id`
- `display_name`
- `predicted_energy_kwh`
- `predicted_cost_eur`
- `predicted_co2_kg`
- `as_of_date`

### `landlord_monthly_summary.json` / `landlord_annual_summary.json`

Landlord-level totals grouped by month/year.

### `tenant_monthly_summary.json` / `tenant_annual_summary.json`

Tenant-level totals grouped by month/year.

### `metrics.json`

Core model metadata plus the demo export config.

Key fields:

- `model_name`
- `validation_cutoff`
- `validation_metrics`
- `baseline_metrics`
- `config`
- `demo_config`

### `contract_manifest.json`

Machine-readable index of generated files and conventions.

## Intended Frontend Usage

- Fake login reads `demo_users.json`
- Landlord views use `landlord_summaries.json` and `property_summaries.json`
- Tenant views use `tenant_summaries.json`
- Detail charts use `daily_forecast.json`, `monthly_summary.json`, and `annual_summary.json`

## Scope Notes

- This is demo-only fake auth, not production authentication
- The backend remains the source of truth for forecasts, cost, CO2, and totals
- Frontend-specific strings and formatting should stay in the frontend
